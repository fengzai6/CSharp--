# 零、环境准备与项目骨架

> 预估时间：1-2 天 | 目标：能创建、运行、调试一个最小 .NET 项目

---

## 为什么先学这一章

如果直接从 C# 语法开始，很容易知道语法但不知道项目怎么跑。先把 SDK、项目文件、命令行和目录结构搞清楚，后面的 ASP.NET Core、EF Core、SignalR 才能顺着学。

本阶段只解决 4 个问题：

1. 机器上装了哪个 .NET SDK
2. `.sln` 和 `.csproj` 是什么
3. 如何创建、运行、调试项目
4. 如何安装 NuGet 包

## 老师提示

这一章不要追求理解所有模板代码。你的目标是把 .NET 项目跑起来、看懂项目文件、知道依赖怎么装。后面遇到编译失败、API 文档打不开、包还原失败时，先回到本章排查 SDK、TargetFramework、环境变量和 NuGet。

## 学习顺序建议

1. 先执行 `dotnet --info`，确认本机 SDK。
2. 再创建 console 项目，确认命令行能编译运行。
3. 然后创建 `.sln` + 多项目结构，理解依赖方向。
4. 最后运行 Web API，并打开开发环境的 API 文档页面。

## 常见误区

- 看到 `.sln` 就以为它等同于项目代码；实际 `.sln` 只是管理多个 `.csproj` 的容器。
- `TargetFramework` 写了 `net10.0`，但本机只有 .NET 8 SDK，导致编译失败。
- 端口写死为 5000；实际端口要以 `dotnet run` 输出为准。
- 还没理解 Api/Core/Infrastructure 依赖方向，就开始堆业务代码。

## 版本选择

学习时优先使用当前机器上的 **LTS SDK**。不要一开始追 preview 版本。

```bash
dotnet --info
dotnet --list-sdks
dotnet --list-runtimes
```

如果机器上有多个 SDK，可以在项目根目录添加 `global.json` 固定版本：

```bash
dotnet new globaljson
```

生成后检查：

```json
{
  "sdk": {
    "version": "10.0.100"
  }
}
```

> 版本号以你本机 `dotnet --list-sdks` 输出为准。学习文档中的语法和 API 以稳定版 SDK 为主。

## 第一个控制台项目

```bash
mkdir csharp-lab
cd csharp-lab
dotnet new console -n HelloCSharp
cd HelloCSharp
dotnet run
```

看到类似输出即可：

```text
Hello, World!
```

### 项目文件

`HelloCSharp.csproj` 类似前端项目里的 `package.json`，记录目标框架、包引用、编译配置。

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net10.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
  </PropertyGroup>
</Project>
```

关键点：

| 配置 | 作用 |
|------|------|
| `TargetFramework` | 目标 .NET 版本 |
| `ImplicitUsings` | 自动引入常用命名空间 |
| `Nullable` | 启用可空引用类型检查 |
| `OutputType` | `Exe` 表示可执行程序 |

## 解决方案与多项目结构

真实项目通常不是单个 `.csproj`，而是一个 `.sln` 管理多个项目。

```bash
mkdir MyApp
cd MyApp
dotnet new sln -n MyApp

dotnet new webapi -n MyApp.Api
dotnet new classlib -n MyApp.Core
dotnet new classlib -n MyApp.Infrastructure

dotnet sln add MyApp.Api/MyApp.Api.csproj
dotnet sln add MyApp.Core/MyApp.Core.csproj
dotnet sln add MyApp.Infrastructure/MyApp.Infrastructure.csproj
```

推荐初学结构：

```text
MyApp/
├── MyApp.sln
├── MyApp.Api/              # HTTP API、Program.cs、Controller/Endpoint
├── MyApp.Core/             # 业务模型、接口、纯业务逻辑
└── MyApp.Infrastructure/   # EF Core、Redis、外部服务实现
```

项目引用：

```bash
dotnet add MyApp.Api/MyApp.Api.csproj reference MyApp.Core/MyApp.Core.csproj
dotnet add MyApp.Api/MyApp.Api.csproj reference MyApp.Infrastructure/MyApp.Infrastructure.csproj
dotnet add MyApp.Infrastructure/MyApp.Infrastructure.csproj reference MyApp.Core/MyApp.Core.csproj
```

依赖方向：

```text
Api -> Core
Api -> Infrastructure
Infrastructure -> Core
Core 不依赖其他项目
```

## 常用命令

| 命令 | 用途 |
|------|------|
| `dotnet new console -n Demo` | 创建控制台项目 |
| `dotnet new webapi -n MyApp.Api` | 创建 Web API 项目 |
| `dotnet restore` | 还原 NuGet 包 |
| `dotnet build` | 编译 |
| `dotnet run --project MyApp.Api` | 运行指定项目 |
| `dotnet watch --project MyApp.Api` | 文件变化后自动重启 |
| `dotnet test` | 运行测试 |
| `dotnet add package <PackageName>` | 安装 NuGet 包 |
| `dotnet remove package <PackageName>` | 移除 NuGet 包 |

## NuGet 包

NuGet 对应 Node.js 生态里的 npm。

安装包：

```bash
dotnet add package FluentValidation
```

安装后 `.csproj` 会出现：

```xml
<ItemGroup>
  <PackageReference Include="FluentValidation" Version="..." />
</ItemGroup>
```

还原依赖：

```bash
dotnet restore
```

## 常见问题

### `dotnet` 命令不存在

说明 SDK 没安装或环境变量没生效。先重新打开终端，再执行：

```bash
dotnet --info
```

### `TargetFramework` 和 SDK 不匹配

例如项目写 `net10.0`，但本机只有 .NET 8 SDK，会编译失败。解决方式：

1. 安装对应 SDK
2. 或把 `.csproj` 的 `TargetFramework` 改成本机已有版本

### Swagger 打不开

先检查终端输出的 URL，再确认是否是 Development 环境。很多模板只在开发环境启用 Swagger。
